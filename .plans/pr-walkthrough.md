# PR Walkthrough — implementation plan

Turn the converged Devin-styled walkthrough prototype into a real feature: a guided,
chapter-by-chapter reading of a PR, generated by an agent from the PR diff.

## Decisions (locked)

- **Sequencing:** thin vertical slice end-to-end (real Codex generation → cache → RPC → real UI), behind a flag.
- **Chapter granularity:** hunk-level. Chapters anchor to `(filePath, oldStart)` hunk refs (the stage-chapters model), not whole files.
- **Prototype:** build a new `ReviewWalkthrough` component, flag-gated. Delete `ReviewWalkthroughPrototype.tsx` + the `?prototypeWalkthrough` route flag at parity.
- **Findings:** reuse the existing line-anchored `ReviewFinding`. Drop the prototype's invented `confidence`/`suggestedFix`/`AnalysisSignal` for v1.
- **"Ask" composer:** reuse the existing `ReviewSidechat` agent, not the prototype's dead AskBar.

## Architecture

```
loadChangeset → patch + patchSignature + headSha           (exists)
   │
   ▼  parseUnifiedDiffHunks(patch) → ParsedHunk[]           (NEW, server)
ReviewSource.generateWalkthrough(input)                     (NEW; mirrors runAgentReview)
   → textGeneration.generateWalkthrough({patch, hunks, prTitle, prBody})  (NEW; runCodexJson + outputSchemaJson)
   → agent clusters hunks → chapters(hunkRefs) + prologue + judgment questions
   → validate hunk coverage (every hunk in exactly one chapter)           (NEW; like stagereview show)
   → staleness guard (patchSignature/headSha)               (copy from runAgentReview)
   → SQLite review_walkthroughs (key: repo, reference, patchSignature; immutable, no TTL)  (NEW table)
   → ReviewUpdateBus push: "pullRequestWalkthrough" tag      (NEW variant)
   │
   ▼  wsRpc review.generateWalkthrough                       (NEW method)
web react-query  ["review","walkthrough",cwd,ref,patchSignature]  staleTime: Infinity   (NEW key)
   │   (key rotates when a push changes patchSignature → fresh fetch; old GCs)
   ▼
ReviewWalkthrough  (NEW; "walkthrough" tab in ReviewPrView, flag-gated)
   ├─ Prologue: motivation/outcome/keyChanges/focusAreas/complexity   (Devin styling ported)
   ├─ Chapters: hunkRefs → subPatchForHunks → FileDiffCard (split/unified, viewed)
   ├─ per-chapter findings → existing ReviewFinding (line-anchored)
   └─ Ask → ReviewSidechat (reuse)
```

## Key reuse (do NOT rebuild)

- Diff render: `FileDiffCard`/`FileDiffSurface` (`apps/web/src/components/chat/FileDiffView.tsx`) + `getRenderablePatch()` (`apps/web/src/lib/diffRendering.ts`) inside `DiffWorkerPoolProvider`. Replaces prototype `DiffSplit`/`DiffCell`.
- Viewed state: `useReviewViewedFiles(target, paths)` (`apps/web/src/components/review/reviewViewedFiles.ts`).
- Data layer: `reviewQueryKeys` + `reviewLoad*QueryOptions` + `applyReviewUpdatedPayload` (`apps/web/src/lib/reviewReactQuery.ts`).
- Server one-shot agent: `runCodexJson({prompt, outputSchemaJson})` via `CodexTextGeneration` / `ProviderTextGeneration` dispatcher; `ReviewSource.runAgentReview` is the template (`apps/server/src/review/Layers/ReviewSource.ts:1585`).
- Cache: `ReviewCacheStore` + migration pattern (`039_ReviewCache.ts`, `review_cache_pr_diff`), signed by `patchSignature`.
- Contracts conventions: `Schema.Struct`/`Schema.Literals`, `TrimmedNonEmptyString`/`PositiveInt`/`NonNegativeInt`, `Review*` naming, value+type same name.

## Risks

- **Mid-flight branch:** `generateReviewFindings` is referenced but not yet declared in `TextGenerationShape`/`CodexTextGeneration`. The review-agent feature is partially landed; P1 may need to finish/coordinate that seam first.
- **Generation latency:** first open triggers a Codex call (seconds); cache makes repeats instant. Needs a real loading state; chapter streaming is a later enhancement.
- **Agent quality:** chapter clustering + prologue quality is prompt-dependent; expect iteration. Hunk-coverage validation catches missing/duplicated hunks.
- **Hunk-level rendering:** `FileDiffCard` renders whole files; per-chapter hunk display needs `subPatchForHunks` to feed it a minimal patch. Validate the reconstructed patch parses.

## Sequencing

P0 → P1 → P2 → P3 → P4. P0 unblocks all. P1 is the bulk (cloned from runAgentReview). P2 is thin. P3 ports the prototype onto real data. P4 guards latency + removes the prototype.
